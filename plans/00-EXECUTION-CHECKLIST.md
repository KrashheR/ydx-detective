# 00 — Порядок выполнения (рабочий чеклист агента)

> **Этот файл — живой трекер.** Агент, работающий по планам: открой этот файл, найди первый
> незакрытый этап сверху вниз, выполни его по указанному плану, **отметь чекбоксы и заполни
> строку «Статус» прямо в этом файле** в том же изменении, что и код.
>
> Правила ведения:
> - `[ ]` → `[x]` только когда пункт реально сделан и проверен, не «почти».
> - Частично сделанный этап: отметь сделанные подпункты, в «Статус» напиши, что осталось и почему
>   остановился (следующий агент продолжит с этого места).
> - Пропуск пункта по решению человека: не ставь `[x]`, пометь `[~]` и причину в «Статус».
> - Не перескакивай через этап с незакрытым блоком **Gate** (ворота верификации) — этапы упорядочены
>   по зависимостям.
> - Обнаружил, что пункт уже реализован в коде: отметь `[x]` с припиской `(уже было в коде)`.
> - **Агент НЕ коммитит и не пушит сам — никогда.** Никаких `git commit` / `git push` / `git add`:
>   по завершении этапа оставь все изменения (код + доки + этот файл) в рабочем дереве и сообщи
>   пользователю, что этап готов к ревью и коммиту. Коммитит только человек.
> - Новый этап начинай только от чистого рабочего дерева (предыдущий этап закоммичен человеком) —
>   иначе изменения этапов перемешаются в одном ревью.

**Сквозные ворота каждого этапа (Gate):** `npm run typecheck` ✅ и `npm test` ✅, обновлён
соответствующий `docs/*.md`. Изменение формы сейва без бампа `saveVersion` + `migrate()` = этап
не закрыт.
