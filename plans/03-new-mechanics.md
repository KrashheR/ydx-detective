## Механика 2.3 — «Досье мошенников» (Картотека)

**Идея.** Каждый разоблачённый мошенник = карточка коллекции: портрет, схема обмана, статистика
игрока по делу. Gold mastery = «полное досье». Долгосрочная цель + реиграбельность.

**Ключевой принцип (инвариант 1):** контент досье — статика в JSON дела; прогресс («открыто ли»,
качество) — **деривативен** из уже персистящихся `stats.completedCaseIds` / `stats.results`
(mastery уже сохраняется — см. `lastResult.mastery`). ⇒ **Форма сейва не меняется, миграция не нужна.**

### Этапы

1. **Данные.** В `Case` (и `caseSchema.ts` — в локстепе) опциональный блок:
   ```ts
   dossier?: {
     suspectName: LocalizedString;
     portraitUrl?: string;          // существующий пайплайн ассетов дел
     scheme: LocalizedString;       // «схема обмана» — 1-2 предложения
     archetype?: string;            // тип схемы: 'staged_theft' | 'inflated_damage' | ...
   }
   ```
   Опциональность = старые дела валидны без изменений. Правило заполнения: досье имеет смысл
   только для дел с `correctDecision: 'reject'` (одобренные честные заявители — не мошенники);
   зафиксировать в `CASE_AUTHORING_GUIDE.json` и `docs/07`.
2. **Селектор.** Чистая функция (например `src/engine/dossierEngine.ts` или селектор в App-слое):
   `(allCases, stats) → DossierEntry[]` — карточка открыта, если дело закрыто верным «reject»;
   уровень карточки = mastery из `stats.results` (bronze/silver/gold). Стора не касается.
3. **UI «Картотека».** Экран/вкладка в стиле архивного каталожного ящика: закрытые карточки —
   силуэт + «Дело не раскрыто», открытые — портрет, имя, схема, штамп mastery. Вход со стола.
   Компонент `components/Dossier.tsx`; i18n ×5 (заголовок, пустое состояние, подписи).
4. **Наполнение.** Пройтись по 38+ шипнутым делам и дописать `dossier` в JSON (контентная работа;
   можно частями — карточки без блока просто не попадают в картотеку). Портреты — по мере
   готовности ассетов, поле опционально.
5. **Связки.** Ачивки «первые 5 досье / полная картотека» через существующий каталог
   `data/achievements.ts` + предикат в `achievementsEngine.ts` (деривативен от stats — ок).
   Аналитика: `dossier_open` (просмотр картотеки), параметр `unlockedCount`.
6. Тесты: dossierEngine (открытость/уровень), Zod-схема; `docs/03` + `docs/07`.

**Оценка:** 2–3 дня кода + контентная работа. **Риск:** низкий.
**Синергия:** карточки досье — строительный блок сюжетных цепочек плана 04 (повторяющийся
антагонист = серия карточек одного архетипа).

## Механика 2.5 — «Кабинет следователя»

**Идея.** Визуальная прокачка стола: штампы, лампа, табличка, грамоты. Материализует уровни
(лечит 1.7), витрина для косметики (IAP путь C — вне скоупа, но закладываем поштучную продажу).

### Этапы

1. **Каталог предметов (статика).** `src/data/officeItems.ts`:
   ```ts
   { id, slot: 'lamp' | 'nameplate' | 'stamp_holder' | 'wall_frame' | ...,
     name: LocalizedString, assetUrl,
     unlock: { kind: 'level', level: number } | { kind: 'achievement', achievementId: string }
           | { kind: 'purchase', productId: string } }   // задел под IAP-путь C
   ```
   Уровневые предметы раскидать по рангам из `gameConfig.progression.ranks` (каждые 2–3 уровня —
   предмет), чтобы у ранга появилась осязаемая награда.
2. **Persisted-состояние.** В `PlayerStats`: `officeSelectedBySlot: Record<string, string>`
   (что выставлено на стол). Разблокировка — деривативна (level/achievements уже в stats),
   хранится только выбор. ⇒ **saveVersion bump + migrate** (пустой объект по умолчанию) —
   объединить с бампом плана 01/05, чтобы не плодить версии.
3. **Store.** Действие `selectOfficeItem(slot, itemId)` с валидацией разблокировки; `persist()`.
4. **UI.** Слои поверх существующего фона стола (предметы = абсолютно позиционированные
   изображения в стиле «бумага и картон»); экран «Кабинет» со слотами и сеткой предметов
   (закрыт — уровень/условие, открыт — «выставить»). При level-up в ResultSheet показывать
   «открыт предмет кабинета». i18n ×5.
5. **1.7-связка.** В момент, когда кабинет в проде, ранги перестают быть чисто числом — можно
   вернуть упоминание уровня в UI разблокировок предметов (но не в гейт дел).
6. Аналитика: `office_open`, `office_item_select` (`itemId`, `slot`, `unlockKind`).
7. Тесты: store-действие, unlock-предикаты; `docs/03`/`docs/05` (новые токены/слои — соблюдать
   дизайн-язык, никакого неона).

**Оценка:** 3–5 дней + ассеты (главный риск — арт в нужном стиле; начать с 5–6 предметов).

## Механика 2.1 — «Связать улики» (cross-reference)

**Идея.** В экспертных делах соединить две карточки, противоречащие друг другу (фото ↔ выписка),
и получить бонус. Чистое детективное удовольствие; усложнение только там, где игрок уже опытен.

**Принципы.** (а) Опциональна на уровне данных — дело без `crossLinks` играется как раньше;
(б) только бонус, никогда штраф за неиспользование (урок 1.5/1.6: механика не должна работать
против игрока); (в) дебютирует не раньше 5-evidence дел (позиция ≥ 24 кампании) — не ломает
инвариант кривой сложности.

### Этапы

1. **Данные + схема.** В `Case`:
   ```ts
   crossLinks?: { a: string; b: string; explanation: LocalizedString }[]
   ```
   Zod: id принадлежат `evidences`, пара уникальна, a ≠ b. Осмысленные пары — только между
   уликами, вместе образующими противоречие.
2. **Сессия.** В `ActiveSession`: `linkedPairs: [string, string][]`. Сессия персистится
   (восстановление прерванного расследования) ⇒ **saveVersion bump + migrate** (пустой массив).
3. **Движок.** В `rewardEngine.evaluateReward` — новый бонусный компонент:
   `linkBonus = round(baseReward * reward.crossLink.bonusPctPerLink/100 * correctLinks)`,
   cap `reward.crossLink.bonusCapPct`. Неверная пара — не штраф, просто не считается
   (ограничить число попыток связывания, напр. `maxLinkAttempts = links.length + 1`, чтобы не
   брутфорсили). Все константы — в `gameConfig.reward.crossLink`. `RewardBreakdown` +
   `linkComponent`.
4. **UI.** Режим «Связать» на столе дела: клик по кнопке → выбор первой карточки → второй →
   красная нить/скрепка между ними (метафора «доски следователя», в рамках дизайн-языка —
   канцелярская нить, никакого неонового «hacker-стиля»). Верная связь — подтверждение + `explanation`;
   неверная — нейтральное «связь не подтвердилась». ResultSheet показывает `linkComponent`.
   i18n ×5. Это самый дорогой пункт — прототипировать на одном деле.
5. **Контент.** Добавить `crossLinks` в 4–6 экспертных дел (L14–16) + сделать механику
   обязательной частью брифа Codex для архивных дел (план 04 — связка!).
6. **Обучение.** Первое дело с crossLinks — мягкая подсказка-оверлей «Заметили связь между
   документами? Свяжите их» (1 раз, флаг в stats → входит в тот же бамп сейва).
7. Аналитика: `cross_link_attempt` (`caseId`, `correct`, `pair`), `linkComponent` в `verdict_submit`.
8. Тесты: rewardEngine (бонус/кап/попытки), Zod, store, UI-режим; `docs/03`, `docs/04`, `docs/07`,
   `CASE_AUTHORING_GUIDE.json`.

**Оценка:** 5–8 дней. **Риск:** средний — UX связывания на тач-устройствах; строгий прототип-гейт:
сначала одно дело за фичефлагом (наличие `crossLinks` в JSON и есть флаг).
