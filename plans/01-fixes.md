## Этап A — 1.5. Пол награды: `total ≥ 0` при верном вердикте

**Проблема.** `evaluateReward` (`src/engine/rewardEngine.ts:155`) считает
`total = positive + bonusComponent - penalty` без пола: при верном вердикте и куче ложных штампов
итог может уйти в минус («я был прав и потерял деньги»).

**Шаги.**
1. `rewardEngine.ts`: `total: Math.max(0, positive + bonusComponent - penalty)`. Поле `penalty`
   в `RewardBreakdown` оставить фактическим (не обрезанным) — ResultSheet продолжает честно
   показывать штраф, просто итог не уходит ниже нуля.
2. Тесты `rewardEngine.test.ts`: кейс «верный вердикт, штраф > позитива → total = 0»; кейс
   «неверный вердикт → по-прежнему 0 без штрафа» (регресс).
3. Проверить `ResultSheet`/`gameStore.submitVerdict`: баланс начисляется из `total`, поэтому
   дополнительных правок стора нет; убедиться, что UI при `total = 0 && penalty > 0` не выглядит
   как баг (при необходимости — строка-пояснение, i18n ×5).
4. `docs/04-economy-progression.md`: зафиксировать пол в формуле.

**Оценка:** ~0.5 дня. **Риск:** нулевой; чисто детерминированная правка.

## Этап B — 1.1. Банкротство → мягкий пол вместо жёсткого гейта

**Текущее поведение.** `balance ≤ bankruptcyThreshold (0)` → `isBankrupt: true`
(`gameStore.ts:641,745,857`), прогрессия блокируется до rewarded-видео `restoreFunds()`
(`gameStore.ts:715`, `restoreFundsTo: 2000`).

**Целевое поведение.** Баланс никогда не блокирует игру. «Восстановить средства за рекламу»
остаётся как добровольный буст при низком балансе.

**Шаги.**
1. `gameConfig.ts`: добавить `economy.lowBalanceOfferThreshold` (например 500) — порог показа
   добровольного оффера. `bankruptcyThreshold`/`restoreFundsTo` оставить (оффер восстанавливает
   до 2000, как раньше).
2. `gameStore.ts`: убрать все места, где `isBankrupt` блокирует старт дела / вердикт; флаг
   `isBankrupt` в `PlayerStats` **не удалять из типа сразу** (см. миграцию), но перестать
   вычислять как гейт — либо переименовать смысл в производный `isLowBalance` на уровне селектора.
   Примечание: после этапа A (пол `total ≥ 0`) баланс может падать только от подсказок/сервисов —
   там уже стоит гейт «не хватает — не купишь» (`buyHint`, `gameStore.ts:442`), т.е. баланс и так
   не уходит ниже 0 → жёсткий пол получается автоматически. Проверить `selectInvestigationService`
   на тот же гейт.
3. UI (`App.tsx` + компонент банкротства): жёсткий блокирующий экран → ненавязчивый оффер
   («Касса пуста? Премия от управления за просмотр ролика») на столе/в ResultSheet при
   `balance < lowBalanceOfferThreshold`. Тексты — i18n ×5 (часть ключей банкротства можно
   переиспользовать/переписать).
4. Персист: форма `PlayerStats` меняется (уходит смысл `isBankrupt`) → **бамп `saveVersion` до 8**,
   в `migrate()` — `isBankrupt: false` для всех старых сейвов (разбанкротить застрявших).
5. Аналитика: goal `bankruptcy` оставить как маркер «баланс упал до 0» (сигнал баланса экономики),
   добавить параметр `blocked: false`; `funds_restore` продолжает работать для оффера.
6. Тесты: store-тесты на «нулевой баланс не блокирует старт дела», миграция v7→v8.
7. Доки: `docs/04` (экономика/банкротство), `docs/06` (миграция сейвов).

**Оценка:** ~1 день. **Риск:** низкий; главное — миграция и вычистка всех веток `isBankrupt` в UI.

## Этап C — 1.7. `requiredLevel`: убрать декоративный гейт из UI

**Проблема.** Реальный замок кампании — строгая последовательность; `requiredLevel` (тир сложности,

которых декоративный.

**Решение (вариант из ревью «убрать из UI, оставить внутренним тиром»).** Уровням реальную функцию
даст «Кабинет следователя» (план 03, механика 2.5) — здесь только чистка UI.

**Шаги.**
1. Найти все места, где закрытое дело объясняется уровнем (компоненты стола/списка дел,
   вероятно `CaseCard`/`Desk`-компоненты; grep по i18n-ключам с «level»/«уровень»). Заменить
   формулировку замка на последовательность: «Сначала завершите предыдущее дело».
2. `caseUnlockEngine.ts` **не трогать**: `(requiredLevel, caseNumber)` остаётся канонической
   сортировкой кампании и инвариантом монотонности (`campaignProgression.test.ts`).
3. Удалить/переписать неиспользуемые i18n-ключи про требование уровня (все 5 языков).

5. Доки: `docs/03-gameplay.md` (гейтинг), при сдвиге инварианта — CLAUDE.md п.7 (сам инвариант

## Этап D — 1.6. Неверный вердикт: разбор «N/M» + градуированный XP

**Проблема.** Неверный вердикт → плоские `wrongVerdictXp = 2` (`gameConfig.progression`), в
ResultSheet нет «что вы упустили». Нашедший все противоречия и ошибившийся кнопкой = тыкнувший наугад.

**Шаги.**
1. **Данные для разбора уже есть.** `lastResult` (`gameStore.ts:649-657`) хранит
   `stampedEvidenceIds` + breakdown; ground truth — в `caseData.evidences[].isContradiction`.
   В ResultSheet (компонент результата) добавить блок при `!verdictCorrect`:
   «Вы нашли N из M противоречий» + список упущенных карточек (заголовок улики + её
   `contradiction`-пояснение из JSON дела — контент уже локализован в кейсах).
2. **Градуированный XP.** `gameConfig.progression`: заменить/дополнить `wrongVerdictXp` на
   `wrongVerdict: { baseXp: 2, proofXpShare: 0.5 }` — XP за неверный вердикт =
   `baseXp + round(xpDifficultyWeight[difficulty] * proofXpShare * (correct / totalContradictions))`.
   Считать в `rankEngine`/месте начисления XP в `submitVerdict` (константы — только в конфиге!).
   Валюта при неверном вердикте по-прежнему 0 — меняем только XP (инвариант «две экономики»).
3. i18n ×5: заголовок разбора, «найдено N из M», «вы упустили».
4. Тесты: rewardEngine/rankEngine — XP-градация (0 найдено → baseXp; всё найдено → максимум);
   UI-тест ResultSheet на блок разбора.
5. Аналитика (синергия с планом 02): в `verdict_submit` уже есть компоненты; добавить
   `correctStamps`/`totalContradictions`, если ещё не шлются.
6. Доки: `docs/04` (формула XP).

**Оценка:** ~1–1.5 дня. **Риск:** низкий; аккуратно с реплеями (`rewardEligible === false` —
тренировка не должна начать сыпать XP; проверить текущее поведение реплеев по XP и сохранить).

## Этап E — 1.4. Подсказка: выбор цели вместо «следующей по порядку»

**Проблема.** `buyHint` раскрывает первую нераскрытую карточку в порядке дела
(`gameStore.ts:397-401`) — игрок платит за случайную с его точки зрения информацию.

**Шаги.**
1. **Store.** Сигнатура `buyHint(caseData, kind, targetEvidenceId?)`: если цель передана и ещё не
   раскрыта — раскрываем её; без цели — прежний fallback (следующая по порядку), чтобы ничего не
   сломать на промежуточных состояниях. Валидация: id принадлежит делу, не раскрыт.
2. **UX (двухшаговый, в дизайн-языке «папки»).** Нажатие «Записка инспектора»/«Опрос свидетелей» →
   режим прицеливания: карточки улик подсвечиваются рамкой «выберите, что проверить», клик по
   карточке = подтверждение покупки (цена уже видна на кнопке подсказки). Esc/клик мимо — отмена.
   Большие хит-таргеты (аудитория 30–60), без анимационного шума. Файлы: `App.tsx` (проброс
   режима), `CaseFile.tsx`/карточки улик, возможно `StampModal.tsx` (читать хирургически!).
3. **Rewarded-ветка (canvass).** Цель выбирается ДО показа ролика; замыкание `reveal` уже
   получает конкретный `targetEvidenceId` (см. `showRewardedAd(() => reveal(0))` —
   `gameStore.ts:455`). Обработать гонку: пока крутился ролик, игрок мог открыть карточку сам →
   в `reveal` уже есть guard на «уже раскрыто» (`gameStore.ts:408`) — просто не списывать/пометить
   canvass использованным осознанно (решить: возвращать попытку или нет; предложение — canvass
   считать использованным, это уже так).
4. i18n ×5: «Выберите улику для проверки», подсказка-тултип режима.
5. Аналитика: в `hint_buy` добавить `targeted: true/false` и `targetIndex` — измерить, вырастет ли
   конверсия подсказок (гипотеза ревью).
6. Тесты: store — раскрытие именно цели, fallback без цели, отказ на уже раскрытую; UI — вход/выход
   из режима прицеливания.
7. Доки: `docs/03-gameplay.md` (подсказки).

**Оценка:** ~1.5–2 дня (основная цена — UX-режим). **Риск:** средний — интеграция в
`StampModal`/`CaseFile`; сессия (`ActiveSession`) не меняет форму → бамп saveVersion не нужен.
