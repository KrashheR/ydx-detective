# План 06 — Мобильный UX: «конфетка» core-loop на телефоне

> Источник: геймдизайн-ревью мобильного флоу от 2026-07-19 (сессия Claude). Ревью писалось по
> коду `MobileDeskMenu.tsx` / `CaseFile.tsx` / `StampModal.tsx` / `VerdictPanel.tsx` /
> `ResultSheet.tsx` / `App.tsx` — перед стартом каждого этапа сверься с исходниками, код мог уйти
> вперёд.
>
> **Диагноз одним абзацем.** Флоу структурно правильный (стол → дело → улики → вердикт), но на
> мобилке он собран как «десктоп, сжатый в колонку»: главные действия не в зоне большого пальца,
> ядро геймплея (сравнение заявления с уликами) разорвано табами, половина обратной связи висит
> на hover-тултипах, которых на таче не существует, а типографика 9–11px противоречит собственному
> манифесту «аудитория 30–60, читаемость превыше всего».

---

## Цели (что значит «стало на порядок удобнее»)

Целевые изменения поведения игрока, измеряемые аналитикой (план 02 уже собирает базу):

| # | Цель | Метрика | Как проверяем |
| - | ---- | ------- | ------------- |
| G1 | Вердикт всегда на расстоянии одного тапа большим пальцем | Время «последний штамп → вердикт» ↓; скроллов на дело ↓ | Замер вручную в mobile-вьюпорте + synthetic playtest (docs/09) |
| G2 | Цикл «открыл улику → прочитал → штамп → следующая» без выхода из просмотра | Тапов на дело с 4–6 уликами: ~10–12 лишних → 0 лишних | Прогон кейса L14 в эмуляции телефона |
| G3 | Игрок видит факты заявления, не покидая улики | Переключений таба statement↔evidence за дело ↓ (новая цель Метрики) | `tab_switch`-счётчик за сессию |
| G4 | Ни одного «мёртвого тапа»: любое недоступное действие объясняет себя | 0 disabled-кнопок с hover-only объяснением в мобильном UI | Код-аудит + ручной прогон |
| G5 | Текст читается аудиторией 30–60 | Ни одного информативного текста < 12px на мобилке | Код-аудит по чеклисту этапа E |

**Не-цели:** редизайн десктопа (менять только мобильные ветки/брейкпоинты `md:`); новые механики;
изменение экономики. Персист-форма не меняется ни в одном этапе → бамп `saveVersion` не нужен
(если по ходу понадобится — это сигнал, что этап вышел за рамки плана, остановись и спроси).

## Как исполнять (для агента Claude Code / Codex)

- Порядок этапов A → F — по убыванию эффекта и возрастанию риска; A и B вместе дают ~80% ценности.
- Один этап = один сеанс работы = один коммит человека. Трекай прогресс в
  `plans/00-EXECUTION-CHECKLIST.md` (этапы 14–16), **сам не коммить**.
- Перед этапом: прочитай «Ключевые файлы» этапа целиком (это компоненты, они небольшие, кроме
  `StampModal.tsx` ~1122 строки — его читай по диапазонам, модальная обёртка — строки 1–210).
- Все новые строки UI — через `src/i18n/ui.ts` (`UIKey` + **все 5 языков**: ru, en, tr, ar, kk).
- Только Tailwind-токены, ни одного hex (этап F чинит существующие нарушения).
- RTL: любая новая мобильная разметка обязана работать при `ar` (`RTL_LANGUAGES`, зеркальные
  стрелки/свайпы — см. существующий паттерн `isRTL` в `CaseFile.tsx`).
- **Верификация каждого этапа (Gate):** `npm run typecheck` ✅, `npm test` ✅, ручной прогон в
  Chrome DevTools mobile-вьюпорте (390×844) минимум одного классического и одного бюджетного дела,
  обновлён `docs/03-gameplay.md` и/или `docs/05-design.md` в том же изменении.
- Скриншоты «до/после» можно снимать `scripts/capture-game-screenshots.mjs` (см. docs/09).

## Этап A — Sticky-вердикт + чистый экран расследования (самое дешёвое / самый большой эффект)

**Проблема.** На мобилке порядок экрана дела: шапка → табы → панель → подсказки → `VerdictPanel` →
**а после него ещё LeftSidebar и RightSidebar целиком** (`App.tsx` — сайдбары не скрыты на мобилке,
а складываются в колонку `order-2`/`order-3`). Игрок штампует улику и должен скроллить мимо
подсказок до вердикта, под которым болтается «хвост» из ранга/стрика/лидерборда.

**Целевой UX.**
1. Во время активного расследования (выбрано дело, вердикта ещё нет) кнопки вердикта живут в
   **закреплённом нижнем action-bar** (`fixed bottom-0`, safe-area-inset, только `md:hidden`):
   Reject / Approve + компактный счётчик «Отмечено: N» (и «🔎 N/M» для бюджетных дел — счётчик
   опенов уже считается в `selectCaseInvestigationGate`).
2. Сайдбары на мобилке с экрана расследования **убрать полностью** (`hidden md:block` на обёртках
   в `App.tsx`). Их контент игрок и так видит на столе (`MobileDeskMenu`) — если чего-то критичного
   там нет (стрик, ранг), добавить компактной строкой в app-bar стола, НЕ обратно на экран дела.
3. Нижний отступ контента = высота бара (контент не должен прятаться под ним).

**Шаги.**
1. `VerdictPanel.tsx`: научить рендериться в двух вариантах (`variant: 'inline' | 'bar'`) или
   вынести мобильный бар отдельным компонентом `VerdictBar.tsx` — что чище по месту, реши по коду;
   логика гейтинга (`canApprove`/`canReject`/`approveBlockedReason`) остаётся пропсами, как сейчас.
2. `CaseFile.tsx`: на мобилке инлайновый `VerdictPanel` заменить на sticky-бар; передать счётчики
   штампов/бюджета (уже приходят пропсами `budget`/`opensRemaining` + `session.selectedEvidenceIds`).
3. `App.tsx`: обёртки Left/RightSidebar → `hidden md:block`; проверить, что мобильный «хвост» исчез.
4. Тост `rejectNeedsProof` не должен перекрываться баром (сейчас `fixed bottom-[22px]` в
   `CaseFile.tsx` — поднять выше бара).
5. i18n: ключ для счётчика «Отмечено» (5 языков), если нет готового.

**Ключевые файлы:** `src/App.tsx`, `src/components/CaseFile.tsx`, `src/components/VerdictPanel.tsx`,
`src/i18n/ui.ts`.
**Оценка:** ~0.5–1 день. **Риск:** низкий; чистая разметка, ноль логики стора.

## Этап B — Просмотр улик: bottom-sheet со sticky-штампом и навигацией «‹ 3/5 ›»

**Проблема.** `StampModal` — центрированный диалог `max-w-[420px]`, целиком `overflow-auto`: на
длинном документе кнопка «Отметить как противоречие» уезжает за скролл. Между уликами нет
навигации: каждая следующая улика = закрыть модалку → попасть по карточке → открыть заново
(~10–12 лишних тапов на дело с 4–6 уликами).

**Целевой UX.**
1. На мобилке модалка становится **полноэкранным bottom-sheet**: шапка (тип + название + ✕) и
   **sticky-футер со штамп-кнопкой** фиксированы, скроллится только тело документа.
2. Внутри просмотра — переключение между уликами: стрелки «‹ 3/5 ›» в футере + горизонтальный
   свайп (порог как у существующего свайпа табов: |dx| ≥ 50 и |dx| > 1.5|dy|). Порядок — как в
   сетке `caseData.evidences`.
3. **Бюджетные дела:** переход на ещё-не-открытую улику = «открытие» → обязан идти через
   существующий гейт `markEvidenceAsViewed` (может вернуть `false` → тост `budgetExhausted`,
   остаёмся на текущей). Не дублируй логику — прокинь колбэк из `App.handleOpenEvidence`.
   Запечатанные (`sealed`) улики в навигации пропускаются или показывают тот же тост — реши по
   простоте, но поведение зафиксируй тестом стора, если добавляешь новую ветку.
4. Десктоп-вид не трогаем (текущий диалог остаётся для `md:`).
5. RTL: стрелки и направление свайпа зеркалятся.

**Шаги.**
1. `StampModal.tsx` (обёртка, ~строки 98–210): раздвоить layout по брейкпоинту; футер с кнопкой
   штампа + стрелками вынести из скролл-области.
2. `App.tsx`: `modalEvidenceId` уже единственный источник правды — добавить хендлер
   `onNavigate(dir)` рядом с `handleOpenEvidence`, переиспользующий его гейт бюджета.
3. Фокус-трап/Escape/возврат фокуса — сохранить (уже реализованы, не сломай).
4. i18n: aria-подписи стрелок (5 языков).
5. Ручной прогон: бюджетное дело — доехать навигацией до незапечатанной и до запечатанной улики.

**Ключевые файлы:** `src/components/StampModal.tsx`, `src/App.tsx`, `src/store/gameStore.ts`
(только чтение гейта), `src/i18n/ui.ts`.
**Оценка:** ~1–1.5 дня. **Риск:** средний — единственный этап, касающийся бюджетного гейта;
покрой навигацию юнит-тестом на уровне хендлера, если выносишь его в чистую функцию.

## Этап C — Заявление всегда под рукой: закреплённая выжимка на табе улик

**Проблема.** Вся игра — «найди противоречие между заявлением и документом», но табы
`statement`/`evidence` показывают строго одно из двух: игрок либо запоминает даты/суммы наизусть,
либо бесконечно свайпает туда-сюда. Это главная фрустрация мобильной сессии.

**Целевой UX.** На табе улик сверху — **сворачиваемая «шпаргалка» заявления** (peek-полоска):
свернуто — одна строка (клиент · сумма · суть в 3–5 слов), тап раскрывает карточку с
`client.meta` (даты, номера полисов — то, с чем сверяют улики) и 1–2 предложениями `claim.story`.
Состояние (свернута/раскрыта) — локальный `useState`, по умолчанию свернута, не персистится.
Шпаргалка видна и в `StampModal`? — **нет**, вне рамок: в sheet этапа B места нет, не раздувай.

**Шаги.**
1. `CaseFile.tsx`: компонент-шпаргалка над `materialsHeader` только для мобильной ветки таба
   `evidence`; данные уже есть в `caseData` (ничего нового в типах/схеме).
2. Выжимку «сути» не генерировать и не добавлять в JSON дел — использовать первые N символов
   `claim.story` с эллипсисом либо просто `title` дела. Никаких изменений `Case`/`caseSchema`.
3. Дизайн: «прикнопленная записка» в языке стола (бумага, токены `paper`/`border`/`text-*`),
   без новых цветов.
4. i18n: подпись «Заявление» / «Свернуть» при необходимости (5 языков).
5. (Опционально, если план 02 уже влит) цель Метрики `tab_switch` — счётчик переключений табов,
   чтобы метрика G3 стала измеримой. Если каталог целей заморожен — пропусти с пометкой в чеклисте.

**Ключевые файлы:** `src/components/CaseFile.tsx`, `src/i18n/ui.ts`, (опц.) `src/services/metrica.ts`.
**Оценка:** ~0.5 дня. **Риск:** низкий.

## Этап D — Ноль «мёртвых тапов»: тач-фидбек вместо hover-тултипов

**Проблема.** На таче hover не существует, а мобильный UI объясняет недоступные действия только
тултипами: заблокированный Approve — `disabled` + `Tooltip` (тап молча съедается — игрок никогда
не узнаёт причину), недоступная «Записка инспектора» — то же, тултипы daily-замка и локов на столе
частично дублированы инлайном, частично нет. Эталон правильного паттерна уже есть в коде: Reject
всегда кликабелен, гейт в хендлере, отказ объясняется тостом (`rejectNeedsProof`).

**Правило этапа (зафиксировать в docs/05):** в мобильном UI недоступное действие — это активная
кнопка в приглушённом стиле, тап по которой показывает причину тостом/инлайном. `disabled` +
hover-only объяснение запрещены.

**Шаги.**
1. `VerdictPanel.tsx`: Approve перевести на паттерн Reject — кликабелен всегда, при `!canApprove`
   тап показывает `approveBlockedReason` тостом (тост-механика уже есть в `CaseFile.handleReject` —
   обобщи её, не копипасть).
2. `CaseFile.tsx`: кнопка «Записка инспектора» при нехватке баланса — не `disabled`, тап →
   существующий текст `tipNoteUnaffordable` тостом.
3. Аудит остальных мобильных `Tooltip`: `MobileDeskMenu` (daily-замок — уже есть `onDailyLocked`,
   проверить; локи дел — причина уже инлайном, тултип можно оставить как прогрессивное улучшение),
   `SpecialArchivesEntry`. Каждому — тач-эквивалент или подтверждение, что информация уже видна
   без hover.
4. Тактильность (дёшево, сюда же): `navigator.vibrate?.(15)` на штамп в `StampModal` и на сабмит
   вердикта. Обёртка-хелпер в одном месте (например `src/utils/haptics.ts`), no-op при отсутствии
   API. Это «удар печати» — главный вкусный момент игры.
5. i18n: новых текстов почти нет (переиспользуются существующие ключи причин).

**Ключевые файлы:** `src/components/VerdictPanel.tsx`, `src/components/CaseFile.tsx`,
`src/components/MobileDeskMenu.tsx`, `src/components/StampModal.tsx`, `docs/05-design.md`.
**Оценка:** ~0.5–1 день. **Риск:** низкий.

## Этап E — Читаемость: типографический проход + мелкие потоки

**Проблема.** Моно-лейблы 9–10px, мета 10–11px, кнопки вердикта 13px uppercase — при манифесте
«30–60 лет, читаемость превыше всего». Плюс два мелких UX-долга.

**Шаги.**
1. Типографика (только мобильные ветки): информативный текст ≥ 12–13px; 9–11px допустимы только
   для чисто декоративных элементов (штампы «СРОЧНО», «КОНФ.», form-номера). Кнопки вердикта —
   15–16px. Пройди `MobileDeskMenu.tsx`, `CaseFile.tsx`, `StampModal.tsx`, `ResultSheet.tsx`;
   составь в чеклисте список правок «было → стало», чтобы ревью было проверяемым.
2. Прогресс расследования на классических делах: в `materialsHeader` показывать «просмотрено N/M»
   (данные: `session.viewedEvidenceIds.length` / `evCount`) — снимает вопрос «я всё посмотрел?»
   перед вердиктом. Бюджетный счётчик «🔎 N/M» остаётся как есть.
3. Режим прицеливания подсказки: баннер «Выберите улику» сделать sticky сверху вьюпорта на мобилке
   + явная кнопка «Отмена» (Esc на телефоне нет); не-таргетируемые карточки слегка приглушать,
   чтобы режим читался мгновенно.
4. Свайп табов: `AnimatePresence mode="wait"` даёт визуальную паузу «улетело — потом влетело» —
   заменить на crossfade/`popLayout`, сравнить на глаз, оставить лучшее.
5. `ResultSheet`: убедиться, что «Следующее дело» (`onNext`) — первичная, самая крупная кнопка,
   «На стол» — вторичная (retention-петля «ещё одно дело»).
6. i18n: «просмотрено», «Отмена» — при отсутствии готовых ключей (5 языков).

**Ключевые файлы:** `src/components/MobileDeskMenu.tsx`, `src/components/CaseFile.tsx`,
`src/components/StampModal.tsx`, `src/components/ResultSheet.tsx`, `src/i18n/ui.ts`.
**Оценка:** ~1 день. **Риск:** низкий; много мелких правок — дели коммит-кандидат на понятные куски
в описании для ревью.

## Этап F — Долг по токенам: `MobileDeskMenu` без hex

**Проблема.** `MobileDeskMenu.tsx` почти целиком построен на инлайновых hex
(`#3a3024`, `#d8b95e`, `rgb(244,233,211)`, …) — прямое нарушение инварианта «только токены»
из CLAUDE.md / docs/05.

**Шаги.**
1. Сверить каждый hex с таблицей токенов в `tailwind.config.js` / docs/05; совпадающие — заменить
   на токен-классы. Отсутствующим в палитре (золото daily-карточки и т.п.) — завести токены
   (расширить `tailwind.config.js`), НЕ оставлять hex и НЕ менять визуал: пиксель-в-пиксель.
2. Тем же проходом — инлайновые hex в мобильных ветках `CaseFile.tsx` (`#b8b1a0`, `#d8d3c7`, …),
   если этап E их ещё не унёс.
3. Скриншот до/после (`scripts/capture-game-screenshots.mjs`) — diff должен быть визуально нулевым.
4. docs/05: новые токены — в таблицу.

**Ключевые файлы:** `src/components/MobileDeskMenu.tsx`, `src/components/CaseFile.tsx`,
`tailwind.config.js`, `docs/05-design.md`.
**Оценка:** ~0.5 дня. **Риск:** низкий (механическая замена), но требует внимательного визуального
сравнения. Удобно делать последним — не конфликтует с A–E по смыслу, но трогает те же строки.
