# План 04 — IAP Путь A: контент (сезонные архивы + связанные дела от Codex)

**Лучший фит для аудитории 30–60: платят за «ещё почитать», не за ускорение.**

**Инфраструктура уже есть — работа меньше, чем кажется:**
- Платёжный адаптер готов: `yandexSDK.ts` — `purchaseProduct()`, `restorePurchasedProductIds()`,
  `getCatalog` (`yandexSDK.ts:400-450`), purchase-goals в Метрике заведены.
- Архивные паки уже существуют: `src/data/cases/archives/`,
  `stats.archivePurchasedPackIds` / `archiveUnlockedCaseIds` /
  `archiveAdUnlockServerDayByPack` в `PlayerStats` + конфиг `specialArchives`.
  ⇒ «Сезонный архив» = существующий архивный пак + два новых свойства: **капельная выдача по
  серверному времени** и **сюжетная связность дел**.

---

## Фаза 1 — Механики связанных дел (движок; предпосылка для контента)

Ревью верно предполагает: для по-настоящему связанных дел нужны доп. механики. Минимальный набор:

1. **Сюжетная цепочка в данных.** В `Case` (+ `caseSchema.ts`):
   ```ts
   storyline?: {
     id: string;                    // 'season-01-podolsky'
     episode: number;               // 1..N, порядок внутри цепочки
     recap?: LocalizedString;       // «В предыдущих делах…» — показ перед стартом эпизода ≥ 2
     epilogue?: LocalizedString;    // добавка в ResultSheet, двигающая арку
   }
   ```
   Гейт: эпизод N доступен после закрытия эпизода N−1 той же цепочки (расширить
   `caseUnlockEngine` — по образцу строгой последовательности кампании; прогресс деривативен из
   `completedCaseIds`, **сейв не меняется**).
2. **Сквозные персонажи через Досье (план 03/2.3).** Общий `dossier.suspectName`/`archetype` у дел
   одной цепочки; карточка досье антагониста «дополняется» с каждым эпизодом (уровни карточки по
   числу закрытых эпизодов). Дёшево и даёт «повторяющегося антагониста» из раздела 5 ревью.
3. **Улики-переносы (carry-over).** Лёгкая версия без движка: эпизод N содержит улику типа
   `document` — «выписка из дела №…» (внутримировая отсылка к прошлому эпизоду). Чистый контент.
   Тяжёлая версия (если Codex-дела попросят): `crossLinks` из плана 03/2.1 как обязательная
   механика финального дела цепочки («связать улики из разных эпизодов» НЕ делаем — только внутри
   одного дела, иначе ломается статичность сессии).
4. **Капельная выдача.** В конфиге архива-сезона: `releaseSchedule: { caseId, unlockAtServerMs }[]`
   (или проще: `dripPerWeek: 2` + `startServerMs`). Проверка — только `getServerTimeMs()`
   (инвариант 3, никакого device clock; оффлайн — показывать «выйдет позже» без точного таймера).
   Закрытая карточка будущего эпизода на столе = тизер («Дело поступит на следующей неделе») —
   и точка для `locked_case_click` из плана 02.
5. Тесты: unlock-цепочка, drip по серверному времени, Zod. Доки: `docs/03`, `docs/07`.

**Оценка фазы:** 3–4 дня.

## Фаза 2 — Продукт и витрина

1. **Товары в Yandex Console:** `season_archive_01` (главный, 10–12 дел), `mini_pack_*` (3 дела),
   `bundle_all_archives` (скидка). Ценовая лестница из ревью.
2. **Store/persist.** Покупка → `purchaseProduct(id)` → в `archivePurchasedPackIds` (поле уже
   есть!). На буте — `restorePurchasedProductIds()` → merge (восстановление на новом устройстве).
   Consume-логику Яндекса проверить: паки = **non-consumable**, не «поглощать» покупку.
3. **UI витрины.** Существующий экран архивов + карточка сезона: обложка «архивная коробка»,
   список эпизодов с датами выхода, цена. `shop_view`/`product_view`/`purchase_*` — уже в GOAL.
4. **Fallback.** Оффлайн/не-Яндекс: витрина скрыта или «недоступно» (payments == null — уже
   обрабатывается), купленное играбельно из локального сейва.
5. i18n ×5 (названия паков, «выйдет через N дней», кнопки покупки). Тесты + `docs/06`/`docs/07`.

**Оценка фазы:** 2–3 дня. **Зависимость:** решения по ценам; фиксы 1.2–1.4 (условие ревью) —
1.2/1.3 уже готовы, 1.4 в плане 01.

## Фаза 3 — Контент: бриф для Codex

Генерацию тематических связанных дел отдаём Codex. Готовый каркас брифа (дополнить перед запуском):

```
Ты — сценарист детективной игры «Где ложь?» про страхового следователя (аудитория 30–60,
тон — приземлённый реализм, без криминала-жести; «бумага, папки, штампы»).

Задача: сезонный архив из 10 связанных дел — сквозная арка одного антагониста.
Сеттинг сезона: [ВЫБРАТЬ, напр. «строительный холдинг и серия поджогов складов»].

Структура арки: эпизоды 1–3 — с виду независимые дела, общая деталь на фоне (одна и та же
фирма-подрядчик / нотариус / оценщик); 4–6 — деталь становится уликой, антагонист получает имя;
7–9 — его схемы усложняются (подставные лица); 10 — «большое дело» против него самого.

Требования к каждому делу (формат — src/data/CASE_AUTHORING_GUIDE.json, отдам файлом):
- 5–6 улик, 2–3 из них — настоящие противоречия (isContradiction: true с полем contradiction);
- difficulty: hard; тип улик — из 10 существующих типов, обязательно использовать
  bank_statement / phone_records / social_media в ≥ половине дел;
- в каждом деле с эпизода 2 — одна улика-отсылка к предыдущему эпизоду (документ/выписка);
- в финальном деле — 2 пары crossLinks (улики, противоречащие друг другу);
- блок dossier (suspectName, scheme, archetype) — у дел с correctDecision: 'reject';
- блок storyline (id сезона, episode, recap 2-3 предложения, epilogue);
- 1–2 дела в сезоне должны быть с correctDecision: 'approve' (честное заявление, которым
  антагонист пытался нас отвлечь) — иначе игрок выучит «в архиве всегда reject»;
- ВСЕ тексты — LocalizedString на 5 языков: ru, en, tr, ar, kk (ru — первичный, остальные —
  качественный перевод, не подстрочник);
- противоречия должны быть находимы ЧЕСТНО: факт-нестыковка между текстом заявления и уликой
  (дата/сумма/гео/объект), не «интуиция».

Выход: 10 JSON-файлов по схеме + один файл season-manifest (порядок, recap'ы, имена эпизодов).
```

Приёмка контента: JSON валиден Zod-схемой (`npm test` прогоняет валидацию),
вычитка ru/en человеком, проверка «противоречие находимо честно» на живом прогоне каждого дела,
регистрация в `caseLoader.ts` и манифесте архива. Важно: архивные дела вне стандартной кампании ⇒
инвариант монотонности кривой (`campaignProgression.test.ts`) их не касается — убедиться, что
тест фильтрует по типу.

**Итого путь A:** ~1.5–2 недели движок+продукт, контент — параллельно через Codex + вычитка.
