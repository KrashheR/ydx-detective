# Промпт для Codex

Скопируй текст ниже и передай Codex вместе с этим архивом.

---

В корень проекта добавлен пакет готового загрузочного экрана для игры «Где ложь? Симулятор детектива».

Файлы пакета находятся в:

`src/components/GameLoader/`

Нужно корректно интегрировать его в существующий React + TypeScript + Vite проект.

## Контекст проекта

- React + TypeScript;
- Zustand со стойким состоянием и восстановлением активной сессии;
- Zod-валидация JSON дел;
- Yandex Games SDK с облачными сейвами и LocalStorage-фолбэком;
- Framer Motion уже используется;
- локали: `ru`, `en`, `tr`, `ar`, `kk`;
- основной контроллер потока находится в `src/App.tsx`;
- не менять игровые механики, экономику, структуру сейва и правила вердикта.

## Задача

1. Изучи текущий процесс запуска приложения, инициализацию Yandex SDK, получение player, гидрацию Zustand, загрузку/валидацию дел и подготовку первого экрана.
2. Подключи `GameLoader` как полноэкранный верхний слой на время реальной загрузки.
3. Не используй фиктивный таймер как источник готовности.
4. Сформируй реальные сигналы типа `BootSignals`:
   - `sdkReady` — SDK готов или успешно выбран dev/offline fallback;
   - `playerReady` — player получен либо определён guest-режим;
   - `saveHydrated` — облачный сейв или LocalStorage восстановлен и применён;
   - `casesValidated` — необходимые JSON дел загружены и прошли Zod;
   - `assetsReady` — загружены критические изображения/шрифты первого экрана.
5. Используй:
   - `calculateBootProgress`;
   - `getBootPhase`;
   - `areBootSignalsReady`;
   - `useSmoothedProgress`.
6. Передавай текущую локаль игры в `GameLoader.locale` без создания второго независимого language state.
7. Когда все сигналы готовы и отображаемый прогресс достиг 100%, покажи фазу `ready` примерно 350 мс, затем плавно скрой loader.
8. Не размонтируй основной App после готовности и не запускай инициализацию повторно.
9. Не блокируй dev/offline режим: при отсутствии Yandex SDK должен сработать существующий фолбэк.
10. Убедись, что ошибки загрузки не оставляют бесконечный loader. Используй существующую обработку ошибок или добавь безопасный fallback с понятным логированием, не подменяя данные.
11. Не жди загрузки всех изображений всей кампании. Предзагрузи только критические ассеты первого экрана.
12. Проверь, что loader имеет больший `z-index`, чем игровой интерфейс, но не ломает рекламный/паузный оверлей после завершения загрузки.
13. Не переписывай стили компонента без необходимости. Он сам импортирует `GameLoader.css` и содержит desktop/mobile WebP-фоны.
14. Удали демонстрационный `example/AppBootExample.tsx` из production imports; файл можно оставить как документацию.

## Ожидаемый результат

- изменённые файлы проекта;
- короткое описание того, откуда берётся каждый boot-сигнал;
- перечисление добавленных или изменённых тестов;
- подтверждение прохождения TypeScript, Vitest и production build;
- отсутствие изменений в механиках дел, reward engine, XP, balance и сейвах.

## Тесты

Добавь или обнови тесты как минимум для следующих сценариев:

1. Loader виден до завершения гидрации.
2. Loader не скрывается, если хотя бы один обязательный сигнал `false`.
3. Loader получает текущую локаль.
4. После полной готовности loader скрывается.
5. Офлайн/dev-фолбэк не зависает на фазе SDK.
6. Повторный рендер не запускает boot-процесс заново.

Сначала изучи существующую архитектуру и переиспользуй имеющиеся флаги/промисы. Не создавай параллельную систему запуска, если необходимые состояния уже существуют.
