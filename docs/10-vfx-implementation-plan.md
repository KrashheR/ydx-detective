## 1. Задача и ожидаемый результат

Нужно сделать игру живее и отзывчивее, не превращая её в яркую мобильную аркаду.
VFX должны усиливать основную фантазию продукта: игрок физически работает с папкой,
документами, уликами и чернильными штампами, а кульминацией становится разоблачение
мошенника или корректное одобрение законной выплаты.

После реализации игрок должен без чтения дополнительного текста понимать:

1. Нажатие принято системой.
2. Улика открыта впервые или уже была просмотрена.
3. Проверка потратила часть бюджета расследования.
4. Штамп поставлен или снят.
5. Вердикт пока нельзя вынести и почему.
6. Вердикт принят и дело закрыто.
7. Из чего сложилась награда, XP и повышение.
8. Какое новое дело или достижение открылось.

Эффекты не должны задерживать игровые действия, блокировать чтение, вызывать
layout shift или ухудшать работу на мобильных устройствах.

## 2. Непереговорные ограничения

### 2.1. Визуальный язык

Разрешено:

- движение и деформация бумажных листов;
- изменение глубины через существующие тени;
- сухие служебные отметки, карандашные линии и чернильные штампы;
- лёгкий бумажный recoil при ударе;
- процедурные эффекты приборов внутри интерактивных улик;
- короткие haptic-паттерны как дополнение к визуальному эффекту;
- очень небольшое количество чернильных точек непосредственно в момент удара.

Запрещено:

- конфетти, монеты, салюты, звёздные взрывы и лутбоксовая подача;
- неон, glow-ауры, sci-fi градиенты и постоянные сияющие контуры;
- сильный screen shake, blur всего экрана и вспышки белым;
- бесконечные декоративные анимации, не сообщающие состояние;
- VFX поверх длинного текста, ухудшающие читаемость;
- новые hardcoded hex/rgb в компонентах;
- bitmap-частицы или видеоролики ради эффектов базового UI.

### 2.2. Игровая честность

До подсказки или финального разбора нельзя визуально показывать истинное значение
`evidence.isContradiction`.

- Штамп игрока выглядит одинаково на правильной и ошибочно отмеченной улике.
- Эффект постановки штампа не может быть зелёным/красным в зависимости от истины.
- Звук и haptic также не должны отличаться для правильной и неправильной улики.
- Реакция на `approve`/`reject` до получения результата должна быть нейтральной.
- Истинный статус разрешено показывать только существующей подсказкой либо в
  `ResultSheet` после `submitVerdict`.

### 2.3. Архитектурные ограничения

- Не хранить VFX-состояния в Zustand, `PlayerStats`, `ActiveSession` или persistence.
- Не менять save shape и `saveVersion` ради анимаций.
- Store выполняет игровое действие сразу. Анимация является реакцией на новое
  состояние и не является условием выполнения мутации.
- Не использовать глобальный event bus. Межэкранную режиссуру координирует
  `App.tsx`; локальный feedback остаётся внутри соответствующего компонента.
- Не добавлять VFX-поля в JSON дел на первом этапе. Использовать существующие
  `EvidenceType`, `EvidenceMeta` и фактическое UI-состояние.
- Не обращаться к portal SDK или внешнему analytics API из UI-компонентов. Все
  измерения проходят через существующую portal-neutral границу
  `src/services/analytics.ts`.
- Весь новый пользовательский текст добавить в `src/i18n/ui.ts` сразу для `ru`,
  `en`, `tr`, `ar`, `kk`. По возможности эффекты должны объяснять уже существующие
  состояния и не требовать новых строк.

## 3. Аудит текущего состояния

Уже реализовано:

- `EvidenceCard` поднимается на desktop hover и слегка уменьшается при tap;
- `StampModal` появляется как модальный лист/bottom-sheet;
- мобильные вкладки заявления и улик сдвигаются горизонтально;
- на документе предусмотрен крупный штамп противоречия;
- `ResultSheet` имеет вход, финальный штамп и count-up награды;
- интерактивные улики содержат локальные процедурные эффекты;
- `tapHaptic()` даёт единичную вибрацию на поддерживаемых устройствах;
- CSS учитывает `prefers-reduced-motion` для CSS transitions/animations.

Проблемы, которые нужно устранить до расширения VFX:

1. `StampModal` использует `animation: stampIn ...`, но `@keyframes stampIn` в
   `src/index.css` отсутствует. Удар штампа фактически не гарантирован.
2. Motion-параметры разбросаны по компонентам и не образуют единой шкалы.
3. `whileTap={{ scale: 0.99 }}` на карточке почти не ощущается на мобильном.
4. Открытие модалки не связано пространственно с исходной карточкой.
5. Изменение бюджета, viewed-state и blocked-state слабо выражено.
6. Вердикт сразу приводит к результату без ясного момента фиксации решения.
7. Элементы `ResultSheet` появляются почти одновременно и конкурируют за внимание.
8. Баланс, XP, повышение, достижение и разблокировка дела не образуют единой
   последовательности награждения.
9. В `ResultSheet` и отдельных renderer-блоках есть inline-цвета. При работе над
   затронутым эффектом перенести нужные цвета на существующие или новые именованные
   Tailwind-токены.
10. Нет тестового профиля качества VFX и формализованного performance budget.

Исходные mobile-скриншоты для сравнения находятся в
`artifacts/mobile-ux-2026-07-19/`. Перед изменениями сохранить новый baseline с теми
же состояниями и viewport.

## 4. Целевая система движения

### 4.1. Новые файлы

Создать минимальный слой `src/vfx/`:

```text
src/vfx/
├── motion.ts              # durations, easing, springs, distances, variants
├── feedback.ts            # именованные haptic-паттерны и safe wrappers
└── motion.test.ts         # инварианты токенов и reduced-motion helpers
```

Не создавать отдельный runtime-движок, очередь частиц или canvas-слой.

### 4.2. Motion tokens

В `src/vfx/motion.ts` завести единственный источник значений. Имена и стартовые
значения:

```ts
export const MOTION = {
  duration: {
    instant: 0.08,
    fast: 0.14,
    base: 0.24,
    slow: 0.36,
    ceremony: 0.72,
  },
  distance: {
    press: 2,
    paper: 16,
    blocked: 6,
    modal: 20,
  },
  ease: {
    out: [0.2, 0.9, 0.3, 1],
    inOut: [0.4, 0, 0.2, 1],
  },
  spring: {
    paper: { type: "spring", stiffness: 300, damping: 26 },
    stamp: { type: "spring", stiffness: 420, damping: 19 },
  },
} as const;
```

Допустимо скорректировать числа после проверки на реальном устройстве. Нельзя
копировать числовые значения обратно в каждый компонент.

Экспортировать общие variants:

- `pressVariants` — press/release кнопок и карточек;
- `paperEnterVariants` — лист или документ входит в рабочую область;
- `modalSheetVariants` — desktop dialog и mobile bottom-sheet;
- `stampImpactVariants` — постановка штампа;
- `stampRemoveVariants` — снятие штампа;
- `blockedVariants` — короткий горизонтальный отказ;
- `rewardItemVariants` — последовательное раскрытие строк результата;
- `unlockVariants` — снятие архивной пломбы/открытие вкладки.

### 4.3. Reduced motion

На React-root добавить `MotionConfig reducedMotion="user"`. В компонентах со
сложной последовательностью использовать `useReducedMotion()`.

В reduced-motion режиме:

- сохранять мгновенную смену цвета, рамки, текста и состояния;
- не использовать scale overshoot, recoil, shake и параллакс;
- свести переходы к opacity длительностью не более 100 мс;
- count-up можно заменить мгновенным финальным числом;
- не скрывать информационные стадии результата;
- haptic остаётся best-effort, но не должен быть единственным feedback.

### 4.4. Haptic-паттерны

Расширить или заменить `tapHaptic()` именованным API, сохранив silent fallback:

```ts
type FeedbackKind = "tap" | "stamp" | "blocked" | "success";
export function playHaptic(kind: FeedbackKind): void;
```

Рекомендуемые паттерны:

| Kind | Паттерн | Использование |
| --- | --- | --- |
| `tap` | `10–15 ms` | обычная кнопка |
| `stamp` | `28–35 ms` | поставить или снять пользовательский штамп |
| `blocked` | `12, 35, 12 ms` | отказ/лимит/нельзя вынести вердикт |
| `success` | `18, 45, 24 ms` | закрытие дела/повышение |

Не вводить отдельный haptic для правильного пользовательского штампа: это раскроет
истину до вердикта.

## 5. Event map и точное поведение

### 5.1. Общая реакция кнопок

Затрагивает: `EvidenceCard`, `VerdictPanel`, кнопки подсказок, навигацию улик,
основные CTA результата и карточки дел.

Алгоритм:

1. На pointer/touch down элемент опускается на `MOTION.distance.press`, масштаб
   меняется максимум до `0.975`, тень уменьшается.
2. Реакция начинается не позднее следующего animation frame.
3. На release элемент возвращается за `fast` duration.
4. Disabled/blocked контрол не имитирует успешный press. Он вызывает локальный
   `blockedVariants` и существующее объяснение.
5. Hover остаётся только для устройств с hover. Mobile feedback не должен
   зависеть от hover.

Критерии приёмки:

- на touch реакция явно видна, но текст не дрожит;
- двойной tap не вызывает двойную игровую мутацию;
- focus-visible не перекрывается анимацией;
- blocked-кнопки сохраняют крупную tappable area.

### 5.2. Открытие дела

Затрагивает: `App.tsx`, `CaseSelect`, `MobileDeskMenu`, `CaseFile`.

Последовательность:

1. Выбранная карточка дела уходит на `pressed`-глубину.
2. Рабочая папка появляется через `paperEnterVariants`: вертикальный ход 12–16 px,
   opacity и небольшое изменение тени.
3. Заголовок и лист заявления не анимировать раздельно длинным stagger; весь вход
   должен занимать не более 360 мс.
4. При возврате к столу применять короткий обратный переход без церемонии.

Не использовать анимацию как замену загрузчику дела. Если JSON подгружается дольше,
состояние загрузки должно оставаться отдельным и доступным.

### 5.3. Переключение «Заявление / Улики»

Затрагивает: `CaseFile.tsx`.

- Сохранить направление свайпа и RTL-логику.
- Снизить амплитуду текущего перехода с полного `100%` до визуально более
  материального сдвига листов, если это не ломает жест swipe.
- Вход сетки улик: только при первом открытии вкладки дать stagger 35–45 мс,
  максимум на первые 6 карточек. При повторном переключении карточки появляются
  как единый лист без повторной церемонии.
- Смена вкладки не должна сбрасывать scroll, hint targeting или viewed-state.

### 5.4. Открытие и закрытие улики

Затрагивает: `EvidenceCard.tsx`, `StampModal.tsx`, `App.tsx`.

Предпочтительный вариант:

- добавить стабильный `layoutId` вида `evidence-${caseId}-${evidence.id}` карточке
  и контейнеру модалки;
- проверить shared-layout переход отдельно на mobile и desktop;
- если shared layout вызывает clipping, скачок scroll или дорогой repaint,
  отказаться от него и использовать согласованный `paperEnterVariants`.

Поведение:

1. Карточка поднимается, тень усиливается.
2. Документ раскрывается в dialog/bottom-sheet за 240–300 мс.
3. Header и body движутся как один физический объект.
4. При переходе к соседней улике меняется только содержимое документа, а не
   размонтируется вся подложка.
5. Закрытие возвращает фокус на исходную карточку.

Первый просмотр:

- после успешного `markEvidenceAsViewed` на карточке появляется небольшая сухая
  отметка/галочка;
- она проигрывается только при переходе `viewed: false → true`;
- повторное открытие не повторяет эффект;
- отметка не должна напоминать правильный verdict и не использует `success` как
  семантику истинности.

### 5.5. Бюджет расследования

Затрагивает: `CaseFile.tsx`, `VerdictPanel.tsx`, возможно отдельный маленький
компонент `InvestigationBudgetIndicator`.

Сделать визуальный индикатор из существующих чисел `budget` и `opensRemaining`.
Не добавлять новое игровое состояние.

При успешном первом открытии:

1. Число/деление слегка утапливается.
2. Одно деление гаснет или получает сухую отметку «использовано».
3. Новое значение становится стабильным не позднее 250 мс.

При последней доступной проверке:

- один раз подчеркнуть индикатор `accent`/`stamp`-чернилами;
- не включать бесконечный pulse;
- не закрывать доступ к уже просмотренным уликам.

При исчерпанном бюджете:

- документ или навигационная кнопка сдвигается назад на 4–6 px;
- проигрывается `blocked` haptic;
- показывается существующий `budgetExhausted` toast;
- текущая улика остаётся открытой;
- VFX не должен создавать впечатление, что бюджет можно купить прямо сейчас.

### 5.6. Targeting подсказки

Затрагивает: `EvidenceCard.tsx`, `CaseFile.tsx`.

Текущий бесконечный box-shadow pulse заменить или смягчить:

- у targetable-карточки один раз рисуется контур карандашом/чернилами;
- после первого прохода остаётся стабильная рамка;
- нет постоянного glow;
- нецелевые карточки приглушаются, но их текст остаётся читаемым;
- выбор карточки вызывает короткую служебную отметку раскрытия;
- отмена `Escape` или кликом вне зоны возвращает карточки за `fast` duration.

Истинный статус после оплаченной/рекламной подсказки можно показать существующими
цветами `stamp` и `success`, потому что механика уже официально раскрыла результат.

### 5.7. Постановка штампа — ключевой эффект

Затрагивает: `StampModal.tsx`, `EvidenceCard.tsx`, `src/vfx/motion.ts`,
`src/vfx/feedback.ts`.

Целевая длительность: 380–480 мс. Последовательность:

| Время | Событие |
| --- | --- |
| `0–70 ms` | CTA утапливается, игровое состояние меняется сразу |
| `50–120 ms` | визуальный штамп появляется чуть крупнее финального размера |
| `110–190 ms` | impact: быстрый scale-down и `stamp` haptic |
| `150–260 ms` | бумага получает recoil 3–4 px и возвращается |
| `170–360 ms` | отпечаток проявляется от центра к краям |
| `200–420 ms` | 2–3 маленькие чернильные точки исчезают |
| `420 ms` | остаётся стабильный штамп без движения |

Технические требования:

- Использовать `AnimatePresence` и Framer variants вместо недостающего CSS
  `stampIn` либо определить keyframes в одном месте. Предпочтительно Framer для
  согласованного reduced motion.
- Штамп получает `key`, зависящий от `evidence.id` и факта постановки, чтобы он
  проигрывался один раз на переход `false → true`.
- Обычный rerender, смена языка и обновление баланса не должны переигрывать удар.
- Recoil применяется к локальному бумажному body, а не ко всему viewport.
- Чернильные точки реализовать DOM/CSS-псевдоэлементами; не подключать particle
  library.
- Финальный отпечаток должен оставаться читаемым на фото, документе, терминальном
  логе и интерактивной улике.
- На карточке сетки corner-stamp появляется коротким scale/opacity, но без второго
  сильного удара после закрытия модалки.

Снятие штампа:

- не повторять impact;
- отпечаток быстро бледнеет/стирается за 140–200 мс;
- CTA возвращается в outline-state;
- использовать тот же нейтральный `stamp` haptic, не success/error.

### 5.8. Заблокированный Reject и Approve

Затрагивает: `VerdictPanel.tsx`, `CaseFile.tsx`.

Reject без штампа:

1. Кнопка делает один цикл `x: 0 → -6 → 5 → 0` не дольше 220 мс.
2. Строка количества отметок получает краткое подчёркивание `stamp`.
3. Показывается существующий `rejectNeedsProof` toast/inline reason.
4. Событие `reject_blocked` по-прежнему отправляется максимум один раз на дело.

Approve до выполнения gate:

- тот же blocked-pattern, но подсвечивается условие просмотра;
- не использовать красный цвет, если причина не является ошибкой игрока;
- tooltip остаётся дополнительным desktop-объяснением, не единственным feedback.

### 5.9. Постановка вердикта

Затрагивает: `VerdictPanel.tsx`, `App.tsx`, `ResultSheet.tsx`.

Игровая мутация `submitVerdict` выполняется сразу после валидного клика. Не ждать
`onAnimationComplete` для расчёта награды.

Визуальная последовательность:

1. Нажатая кнопка фиксируется в pressed-state; обе verdict-кнопки перестают
   принимать повторные клики.
2. Поверх текущего дела появляется затемняющая подложка результата.
3. Лист результата входит снизу/из глубины, будто заключение положили на папку.
4. До показа `verdictCorrect` использовать только нейтральное движение бумаги.
5. Финальный служебный штамп сообщает успех или ошибку.

Не вводить отдельную artificial delay перед показом результата. Вся входная
режиссура должна быть частью `ResultSheet`, уже имеющего рассчитанный result.

### 5.10. ResultSheet и награда

Затрагивает: `ResultSheet.tsx`, `useCountUp.ts`, `RightSidebar.tsx`.

Разбить результат на стадии. Рекомендуемый timeline:

| Момент | Что появляется |
| --- | --- |
| `0–180 ms` | лист результата и затемнение |
| `120–480 ms` | финальный штамп |
| `300–650 ms` | итоговая формулировка по делу |
| `450–1050 ms` | count-up гонорара/убытка |
| `650–1100 ms` | mastery и бонус идеального дела |
| `800–1250 ms` | точность и строки breakdown |
| `1000–1450 ms` | XP и бюджетная эффективность |
| `после 1200 ms` | повышение/достижения как отдельная очередь |

Правила:

- Пользователь может нажать CTA сразу после появления dialog; анимация не должна
  удерживать его до конца timeline.
- Строки появляются сверху вниз, но уже выделенное экранное место не меняет
  layout. Использовать opacity/transform, а не поздний mount, где это вызывает
  скачок высоты.
- Count-up запускается один раз на новый `caseId/result`.
- Double reward анимирует только изменившуюся бухгалтерскую строку и итог, а не
  заново весь экран.
- Ошибка не должна иметь более долгую и тяжёлую анимацию, чем успех.
- Удалить затронутые hardcoded цвета и заменить токенами.
- На мобильном проверить, что основной вывод, сумма и первый CTA видны без
  непредсказуемого overlay сквозь фон.

Синхронизация с боковой панелью:

- после закрытия `ResultSheet` баланс и XP в `RightSidebar` получают короткий
  count/change highlight;
- не анимировать число бесконечно при каждом rerender;
- не рисовать летящие монеты между modal и sidebar.

### 5.11. Повышение, достижение, новое дело

Затрагивает: `ResultSheet.tsx`, `RightSidebar.tsx`, `AchievementsModal.tsx`,
`CaseSelect.tsx`, `MobileDeskMenu.tsx`.

Очередь при нескольких событиях:

1. Награда и XP.
2. Повышение ранга.
3. Новое достижение.
4. После возврата на стол — разблокировка следующего дела.

Визуальная метафора:

- повышение — новый штамп в личном деле;
- достижение — архивная карточка/нашивка, укладываемая на лист;
- открытие дела — снятие пломбы и восстановление насыщенности вкладки.

Каждая стадия не более 500–700 мс. Если разблокировано несколько достижений,
показать компактный список после одного общего входа, а не серию полноэкранных
попапов.

### 5.12. Интерактивные улики

Затрагивает: `InteractiveEvidence.tsx`, интерактивный участок `StampModal.tsx`.

Не переписывать существующие механики. Добавить общий success-feedback после
перехода `analysisCompleted: false → true`:

1. Активный прибор/сканер стабилизируется или выключается.
2. Найденная область остаётся видимой.
3. Заключение появляется как служебная отметка анализа.
4. Разблокировка кнопки штампа получает короткий локальный highlight.

Допустимые индивидуальные акценты:

- `thermal_scan` — температура стабилизируется, ring мягко фиксируется на зоне;
- `surface_reveal` — маска перестаёт реагировать, контур находки дочерчивается;
- `shadow_time_check` — геометрическая линия фиксируется карандашной отметкой;
- `document_scan` — найденные поля получают сухие рамки проверки;
- `seal_match` — совпавшие формы слегка защёлкиваются без bounce-игрушечности.

Решение интерактивной улики не должно автоматически ставить штамп противоречия.

## 8. Тестовая стратегия

### 8.1. Unit/component tests

Обязательные сценарии:

1. `prefers-reduced-motion` не скрывает финальные состояния.
2. Постановка штампа вызывает store action ровно один раз.
3. Rerender с тем же `stamped=true` не создаёт новый impact.
4. Снятие штампа не проигрывает success/impact sequence.
5. Заблокированный Reject не вызывает `submitVerdict`.
6. Быстрый двойной клик verdict вызывает одну отправку.
7. Новая улика уменьшает отображаемый budget; повторное открытие — нет.
8. Закрытие modal возвращает focus на исходную карточку.
9. Переход между уликами сохраняет modal и правильную позицию.
10. RTL меняет направление свайпа и стрелок, но не timeline.
11. Result stages отображают все данные при reduced motion.
12. Double reward не переигрывает весь `ResultSheet`.

Не тестировать точные промежуточные CSS transform по миллисекундам. Тестировать
события, lifecycle, финальные состояния, отсутствие повторной мутации и
accessibility.

### 8.2. Visual QA matrix

Минимальная матрица:

| Surface | Mobile | Desktop | RTL | Reduced motion |
| --- | --- | --- | --- | --- |
| Стол/открытие дела | ✓ | ✓ | ✓ | ✓ |
| Statement ↔ evidence | ✓ | ✓ | ✓ | ✓ |
| Photo/document/log evidence | ✓ | ✓ | один тип | ✓ |
| Interactive evidence | ✓ | ✓ | ✓ | ✓ |
| Stamp add/remove | ✓ | ✓ | ✓ | ✓ |
| Budget exhausted | ✓ | ✓ | ✓ | ✓ |
| Correct/wrong result | ✓ | ✓ | ✓ | ✓ |
| Rank/achievement unlock | ✓ | ✓ | достаточно одного RTL | ✓ |

Viewports:

- `390×844` touch;
- короткий mobile landscape;
- tablet portrait;
- desktop `1365×768`;
- desktop `1440×900`.

### 8.3. Performance budget

- Визуальная реакция на input начинается не позднее 80 мс.
- Цель — 60 FPS; p95 frame time не более 20 мс на целевом mobile-профиле.
- Не более 6–8 одновременно анимируемых элементов во время обычного действия.
- Не анимировать `width`, `height`, `top`, `left`, если можно использовать
  `transform` и `opacity`.
- Избегать больших blur/filter и анимации box-shadow на весь viewport.
- Не создавать постоянный `requestAnimationFrame` вне существующих интерактивных
  механик.
- VFX не создаёт CLS и не меняет scroll position.
- Новая VFX-инфраструктура не должна добавлять крупные raster assets.

## 9. Аналитика и проверка гипотезы

Не отправлять отдельное событие для каждой фазы анимации. Использовать существующие
события игрового пути:

- `evidence_open` / `evidence_close`;
- `evidence_stamp`;
- `reject_blocked`;
- `budget_exhausted`;
- `verdict_submit`;
- `result_view` / `result_action`;
- `case_complete`;
- `investigation_interrupt` / `investigation_resume`.

Для A/B добавить в общий analytics context параметр `vfxVariant` (`control` или
`material_feedback_v1`). Не менять транспорт и не обращаться к Metrica напрямую.

Главные продуктовые показатели:

| Метрика | Ожидаемое направление |
| --- | --- |
| Доля `reject_blocked` | `−20–30%` |
| Прерывания расследования | `−10–15%` |
| Завершение первого дела | `+5–10 п.п.` |
| Переход с результата к следующему делу | `+5 п.п.` |
| Время до первого открытия улики | снижение без роста misclicks |
| Точность штампов / false stamps | не хуже control, желательно лучше |
| Proof precision | не хуже control |

Нельзя считать рост количества штампов самостоятельным успехом: яркий feedback не
должен провоцировать игрока штамповать все карточки.

Качественные вопросы плейтеста:

1. Что произошло после нажатия на улику?
2. Понял ли игрок, что проверка была потрачена?
3. Понимает ли игрок разницу между «просмотрено» и «отмечено как противоречие»?
4. Что означает финальный штамп?
5. Из чего сложилась награда?
6. Заметил ли игрок повышение и открытие следующего дела?
7. Не мешают ли эффекты чтению и концентрации?

Цель: минимум 85% участников правильно объясняют последствия штампа, расхода
проверки и вердикта.

## 10. Definition of Done всей инициативы

Инициатива считается завершённой, когда одновременно выполнено следующее:

- [ ] Все P0-события из event map имеют визуальную реакцию.
- [ ] Удар штампа является единым, переиспользуемым и одноразовым эффектом.
- [ ] Ни один pre-verdict VFX не раскрывает `isContradiction`.
- [ ] VFX не добавлены в store, persistence или case JSON.
- [ ] Нет новых hardcoded цветов и пользовательских строк.
- [ ] Reduced motion сохраняет все информационные состояния.
- [ ] Mobile, desktop, RTL и keyboard navigation проверены.
- [ ] Performance budget выдержан на целевом mobile-профиле.
- [ ] `npm run typecheck` проходит.
- [ ] `npm test` проходит.
- [ ] Visual QA artifacts приложены к изменению.
- [ ] Метрики имеют baseline и способ сравнения control/variant.
- [ ] Изменённые механики и дизайн отражены в `03-gameplay.md` и
  `05-design.md`; аналитика/проверка обновлены в `06`/`08`, если затронуты.
