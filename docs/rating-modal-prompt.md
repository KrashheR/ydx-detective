# Промпт для Claude Design — модалка «Оцените игру» (Яндекс.Игры)

> Скопируй весь этот файл как промпт. Это бриф уровня senior game designer: даёт контекст,
> технические ограничения платформы, концепцию, UX-логику триггеров и точную UI-спецификацию.
> Цель — сгенерировать дизайн (и, по возможности, React-компонент) модалки-приглашения оценить игру.

---

## 0. Роль

Действуй как **senior game designer + product designer**, который выпускает игры на **Яндекс.Играх**.
Твоя задача — спроектировать ненавязчивую, конверсионную и **не нарушающую правила платформы**
модалку «Понравилась игра? Поставь 5 звёзд», вписанную в визуальный язык конкретного продукта.

## 1. Контекст продукта (жёсткие рамки дизайна)

**«Где ложь? Симулятор детектива»** — data-driven игра про расследование страховых дел.
Игрок изучает дело, открывает карточки-улики, штампует противоречия и выносит вердикт
approve/reject. Аудитория **30–60 лет**.

Главная метафора экрана: **«папка с расследованием, разложенная на столе детектива»** —
бумажные документы, официальные бланки, чернильные штампы, архивные папки. Это **не** яркая
мобильная казуалка. Приоритет: высокая читаемость текста, минимализм, чёткая иерархия,
крупные зоны нажатия.

Ключевая эмоция, которую обслуживает игра: **«я заметил ложь и разоблачил мошенника»**.
Модалка должна ловить именно этот момент гордости — не выпрашивать, а как бы «поощрять».

## 2. Технические факты Яндекс.Игр (ОБЯЗАТЕЛЬНО учесть — read first)

Рейтинг на Яндекс.Играх ставится через **нативный диалог платформы**, а не через нашу вёрстку.

```js
const { value, reason } = await ysdk.feedback.canReview();
if (value) {
  const { feedbackSent } = await ysdk.feedback.requestReview(); // откроет НАТИВНЫЙ поп-ап Яндекса
} else {
  // reason: 'NO_AUTH' | 'GAME_RATED' | 'REVIEW_ALREADY_REQUESTED'
  //       | 'REVIEW_WAS_REQUESTED' | 'UNKNOWN'
}
```

**Что из этого следует для дизайна (критично):**

1. **Звёзды и сам выбор оценки рисует Яндекс, а не мы.** Наша модалка — это **soft-ask**
   (мягкое приглашение): «Нравится игра?». Стилизовать нативный диалог нельзя.
2. **Паттерн в два шага** (индустриальный стандарт, снижает риск негативных отзывов и не
   «сжигает» единственный показ на незаинтересованных):
   - **Шаг 1 — наша модалка** в стиле папки: «Понравилось расследование? Оцените работу
     детектива». Кнопки: **«Оценить» (accent)** и **«Не сейчас» (вторичная)**.
   - **Шаг 2** — по нажатию «Оценить» вызываем `requestReview()` → откроется нативный
     поп-ап Яндекса со звёздами. `feedbackSent` сообщает лишь факт отправки, **не оценку**.
3. **`requestReview()` можно вызывать только 1 раз за сессию.** Всегда сначала `canReview()`.
4. **Перед показом нашей soft-ask модалки тоже вызвать `canReview()`** — если `value === false`
   (уже оценил / не авторизован / уже спрашивали), **модалку не показывать вовсе**. Не теряем
   доверие, показывая приглашение тому, кто уже оценил.

**Правила / риск банов и ворнингов модерации:**

- ⚠️ **Нельзя давать игровую награду за оценку.** Технически `feedbackSent` не возвращает число
  звёзд, поэтому «5 звёзд = бонус» невозможно честно реализовать, а «награда за факт отзыва»
  — это incentivized review и риск претензий модерации. **Модалка ничего не дарит за оценку.**
- ⚠️ Не спамить: 1 показ нативного диалога за сессию + наша частотная политика (см. §4).
- ✅ Допустимо показывать soft-ask после позитивного момента — это норма платформы.

## 3. Концепция (senior GD)

**Название макета:** «Благодарность от Департамента» / «Отзыв инспектора».

Визуально модалка — **официальный бланк обратной связи**, лёгший поверх стола: лист `paper`
с печатью-шапкой «ДЕПАРТАМЕНТ РАССЛЕДОВАНИЙ», короткий машинописный текст и место под подпись.
Тон — уважительный и слегка ведомственный, под стать аудитории 30–60 и метафоре игры. Это не
«дай 5 звёзд плиз», а «ваша работа отмечена — порекомендуете коллегам?».

Звёзды в нашей модалке — **декоративная/CTA-аффорданс** (ряд из 5 «чернильных» звёзд как
визуальный якорь на кнопке/над ней), но **фактический ввод оценки происходит в нативном
диалоге Яндекса**. Не имитировать интерактивный рейтинг, который никуда не ведёт.

## 4. UX-логика: когда, где и как часто показывать

**Момент (peak-end):** ловим пик гордости. Показывать **на/после `ResultSheet`** (экран итога
дела), и только когда дело закрыто «красиво»:

- вердикт **верный** (`decision === correctDecision`), **и**
- итоговая награда **положительная**, **и** в идеале совпало с одним из усиливающих событий:
  закрыто **daily-дело**, **повышение ранга**, или **разблокирована ачивка**.

**Анти-триггеры (никогда не показывать):**

- на первом-втором деле — дать игроку «прожить» цикл (порог: **≥ 3 закрытых дела**);
- при банкротстве / отрицательной награде / неверном вердикте;
- если игра на паузе (открыта реклама), идёт другой модал, или это самый первый запуск.

**Частотная политика (хранить в runtime-состоянии, не в Case):**

- максимум **1 показ soft-ask за игровую сессию**;
- если игрок нажал «Не сейчас» — отложить минимум на **N закрытых дел** (предложить 3) и/или
  следующую сессию; счётчик отказов `dismissCount`;
- после **2–3 отказов** больше не показывать (`suppressed = true`);
- если `canReview()` вернул `GAME_RATED`/`REVIEW_WAS_REQUESTED` — `suppressed = true` навсегда.

## 5. UI-спецификация

**Тип:** центрированный модал поверх затемнённого стола (overlay `bg/80`), ширина ~`max-w-[420px]`,
лист `paper`, скругление как у документов, `shadow-lift`. Появление — Framer Motion: лист
«ложится» на стол (lift + лёгкий поворот ±1°, fade overlay). Закрытие — обратно.

**Состав (сверху вниз):**

1. **Шапка-печать** — сериф, `ink`, разрядка букв: «ДЕПАРТАМЕНТ РАССЛЕДОВАНИЙ» + тонкая линейка.
2. **Заголовок** (`font-serif`, крупный): «Дело раскрыто. Отличная работа, детектив.»
3. **Тело** (`font-sans`, читаемый размер): «Ваша внимательность помогает разоблачать мошенников.
   Понравилась игра? Оцените её — это поможет другим найти расследование.»
4. **Ряд из 5 «чернильных» звёзд** — декоративный якорь (цвет `gold`), без ложной интерактивности.
5. **Кнопки** (крупные зоны нажатия, ≥ 44px):
   - **Основная** `accent`: «Оценить» → вызывает `requestReview()`.
   - **Вторичная** (контур/текст, читаемая, не спрятанная): «Не сейчас».
6. (Опц.) мелкая третичная ссылка «Больше не предлагать» — для уважения к игроку.

**Состояния:**

- `idle` (показана), `submitting` (после «Оценить» — короткий лоадер пока открывается нативный
  диалог), `done`/`dismissed` (закрытие с анимацией). Ошибку `requestReview()` глотать молча,
  модалку закрывать — не блокировать игру.

**Доступность / RTL:**

- фокус-трап, закрытие по `Esc` и по клику на overlay (= «Не сейчас»), `aria-modal`, видимый фокус;
- контраст текста на `paper` достаточный (ink); кнопки — крупный таргет;
- **поддержать RTL**: в `SUPPORTED_LANGUAGES` (`src/types/index.ts`) есть RTL-язык `ar` —
  зеркалить раскладку (`dir`), не ломать выравнивание печати/подписи.

## 6. Тексты и локализация

Игра RU-first, но строки **должны быть заведены для ВСЕХ языков** из `SUPPORTED_LANGUAGES`
(`src/i18n/ui.ts`). Zod-схема строгая — пропуск перевода падает на загрузке. Ниже RU/EN как
референс тона; остальные языки добей в том же ключе.

| Ключ                    | RU                                                                                   | EN                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| `rate_eyebrow`          | ДЕПАРТАМЕНТ РАССЛЕДОВАНИЙ                                                             | DEPARTMENT OF INVESTIGATIONS                                                  |
| `rate_title`            | Дело раскрыто. Отличная работа, детектив.                                            | Case closed. Excellent work, detective.                                       |

## 7. Точки интеграции в существующий код (для имплементации)

> Архитектурное правило проекта: **static case data ↔ runtime player state строго разделены**.
> Состояние модалки — это **runtime**, его место в Zustand-сторе/`PersistedState`, НЕ в `Case`.

1. **`src/services/yandexSDK.ts`** — добавить адаптер (по образцу существующих обёрток, с graceful
   fallback в offline): расширить типы `YandexSDK` полем `feedback`, экспортировать
   `canRequestReview(): Promise<boolean>` (внутри `canReview()`, в offline → `false`) и
   `requestReview(): Promise<boolean>` (возвращает `feedbackSent`). SDK трогаем **только** здесь.
2. **`src/store/gameStore.ts` + `PersistedState`** — runtime-слайс `review`:
   `{ asked: boolean; dismissCount: number; suppressed: boolean; lastAskedCaseCount: number }`.
   Логика триггера (§4) — в действии закрытия дела/после вердикта. **Бамп `GAME_CONFIG.saveVersion`
   и расширение `migrate()`** в `src/services/persistence.ts` для нового поля (бэкфилл дефолтов).
3. **`src/App.tsx`** — единственный, кто дёргает действия стора: после показа `ResultSheet`, если
   стор-селектор «можно показать рейтинг» → отрисовать модалку. По «Оценить» → `requestReview()`.
4. **`src/components/RatingModal.tsx`** (новый, **презентационный**) — пропсы:
   `onRate`, `onDismiss`, `onNeverAsk?`. Никакой логики SDK/частоты внутри — только вид и колбэки.
5. **`src/i18n/ui.ts`** — ключи из §6 на все языки.
6. Тюнинг порогов (мин. дел, кулдаун, лимит отказов) — в `src/config/gameConfig.ts`, не в движке.

### Напоминание имплементатору (вне макета)

Реальная оценка собирается **нативным диалогом Яндекса**; наша модалка лишь приглашает. Не
имитируй ввод звёзд, не награждай за отзыв, не показывай приглашение, если `canReview()` дал
`false`. Это и про UX-доверие, и про прохождение модерации Яндекс.Игр.
