# 09 · Синтетическое плейтестирование

> **Ключевые файлы:** `playtest/personas.json`, `playtest/scenario.json`,
> `scripts/synthetic-playtest.mjs`, `playtest/README.md`.

## Назначение

Синтетический плейтест формирует UX-гипотезы до исследования с реальными людьми. Десять LLM-агентов
работают с настоящим интерфейсом через независимые Playwright-контексты, фиксируют впечатления по
шагам и выпускают индивидуальные отчеты. После завершения оркестратор создает сводку повторяющихся
сигналов и план их проверки.

Система не считается пользовательским исследованием и не дает статистически значимых результатов.
Возраст и другие свойства персоны задают контекст поведения, но не являются доказанной причиной
наблюдаемой проблемы.

## Слои

- **Статическая выборка:** `personas.json` содержит ровно 10 профилей 18–75 лет с разной цифровой
  грамотностью, игровым опытом, устройствами и требованиями доступности.
- **Протокол:** `scenario.json` задает цель первой сессии, вопросы наблюдения и условия остановки.
- **Оркестратор:** `synthetic-playtest.mjs` получает скриншот, запрашивает одно структурированное
  действие, проверяет координаты и выполняет его в Playwright.
- **Доказательства:** каждый шаг сохраняется как JSONL, экран — как PNG. Индивидуальные проблемы
  обязаны ссылаться на номера шагов.
- **Отчеты:** JSON является машиночитаемым источником, Markdown — представлением для команды.

Инструмент живет вне `src/` и не меняет runtime игры, store, сохранения или контент дел. Каждый агент
получает чистые cookies и LocalStorage; параллельные сессии не разделяют прогресс.

## Эксплуатация

```bash
npm run playtest:validate
npm run dev -- --host 127.0.0.1
npm run playtest:run -- --url=http://127.0.0.1:5173 --parallel=2
```

Для API-запросов обязательны `OPENAI_API_KEY` и явно выбранный `PLAYTEST_MODEL`. Артефакты пишутся в
`artifacts/playtests/<run-id>/` и исключены из Git. Полный формат запуска, структура каталогов и шкала
severity описаны в `playtest/README.md`.

## Правила изменения

- Изменение состава персон требует сохранения диапазона возрастов и разнообразия устройств/опыта.
- Изменение сценария требует нового `id`, чтобы результаты разных протоколов не смешивались.
- Новое поле отчета добавляется одновременно в JSON Schema и Markdown-рендерер.
- Перед полным прогоном запускается `npm run playtest:validate`, затем одна персона.
- Решения по продукту подтверждаются аналитикой или тестом с реальными представителями сегмента.
